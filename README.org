#+TITLE: MadeManifest
#+OPTIONS: toc:2 num:nil

MadeManifest is a deterministic calculation engine for astrology, Human
Design, and Gene Keys.  The project is on its way from a Golden Test
proof-of-capability to the Trinity Engine v1 runtime contract described
in [[file:specifications/trinity/trinity.org][specifications/trinity/trinity.org]].

The Go engine runs as a command-line tool and as an HTTP service.  The
service is packaged for Docker and Kubernetes, and a shared test
harness exercises it in all three runtimes.

* Go Engine Packages

Phase 12 retired the entire PoC code path.  The packages below are
the trinity-only surface the engine ships today.

=src/mademanifest-engine/pkg/=:

- =canon= – compiled-in canonical constants (=GateOrder=, =SignOrder=, =ChannelTable=, =CenterOrder=, etc.) and version pins.  =SelfCheck()= validates them at boot; =AssertGateSequenceFileMatchesGateOrder= is a deployment / CI tool for verifying any external JSON gate-sequence fixture against the compiled canon.
- =astronomy= – local time → UTC, UTC → Julian Day.
- =ephemeris=, =sweph= – Swiss Ephemeris (cgo) wrappers, pinned to 2.10.03.  =ResolvedEphePath()= surfaces the deployment-resolved data directory for =/version=; =ValidateEphePath()= is a boot-time gate.
- =hd/calc= – Phase 5+ canonical Trinity calculation primitives.  Holds =SolveDesignTime= (pure bisection, backward-only, all canon constants compiled in) and =MapToGateLine= (mandala anchor 277.5°, start-inclusive / end-exclusive interval rule).
- =hd/structure= – Phase 7 structural derivations.  =Compute(personality, design)= produces channels (lex-sorted), centers (canon order), definition (5-class union-find), type (5-way decision tree), authority (8-way priority), profile, and incarnation_cross.
- =trinity/input= – Phase 2 Trinity input validator with full Document 04 rules.
- =trinity/output= – Phase 3 envelope types (=SuccessEnvelope=, =ErrorEnvelope=) with canon-ordered JSON keys.
- =trinity/astro= – Phase 4 Trinity astrology pipeline (signs, houses, 13 objects, angles).
- =trinity/hd= – Trinity Human Design pipeline.  =ComputeDesignTime= (Phase 5) drives the bisection solver; =ComputeActivations= (Phase 6) builds personality + design activation arrays from SE_TRUE_NODE-policy snapshots fed through the canonical mandala mapping.
- =trinity/genekeys= – Phase 8 Gene Keys derivation.  =Compute(personality, design)= reads the four HD pillar activations and emits the canonical four positions (life_work / evolution / radiance / purpose).
- =golden= – Phase 11 Golden Test Pack runner.  =LoadFixtures= walks =src/golden/trinity/<category>/<name>/{input,expected}.json=, enforces canon minimums per category, and returns deterministic =Fixture= records.  =CompareSuccess= / =CompareError= drive each fixture's response through the Trinity comparison rules (semantic JSON equality for success minus metadata; =error_type= + envelope shape for errors per A4).
- =httpservice= – =/healthz=, =/version=, and =/manifest= HTTP handlers (Phase 10 contract).

* Build And Run

** Prerequisites

- macOS or Linux
- Go 1.22 (see =src/env/GO_VERSION.txt=)
- A C toolchain for cgo
- Docker (for the container and Kubernetes harnesses)
- =kind= and =kubectl= (for the Kubernetes harness)

** Swiss Ephemeris

Build and install Swiss Ephemeris 2.10.03 and its data files (one-time
step; uses =sudo= for =rsync= into =/usr/local/share/swisseph=):

#+begin_src bash
make -C src swisseph-install swisseph-install-data
#+end_src

** Compile And Run

Phase 12 retired the file-based PoC CLI.  =cmd/httpserver= is now
the only buildable entry point.  The Makefile target =compile=
emits the binary at =src/mademanifest-engine/mademanifest-engine=:

#+begin_src bash
make -C src compile
#+end_src

Run it with sensible defaults:

#+begin_src bash
cd src/mademanifest-engine
PORT=8080 \
SE_EPHE_PATH=/usr/local/share/swisseph \
./mademanifest-engine
#+end_src

Endpoints (full status-code policy in =doc/user-manual.md=):
- =GET /healthz= – liveness probe (body =={"status":"ok"}==)
- =GET /version= – pinned canon versions + =ephe_path_resolved=
- =POST /manifest= – Trinity payload in, Trinity envelope out

Drive it with the canonical Schiedam fixture from the Trinity pack:

#+begin_src bash
src/scripts/request_cloud_service.sh \
    http://localhost:8080/manifest \
    src/out/cloud-service-output.json
#+end_src

The Makefile aliases =run=, =diff=, and =test-integration= now
delegate to =test-integration-local=, which runs the full Phase 11
golden pack against a freshly-built local subprocess.

** HTTP Service (Docker)

Build the production image:

#+begin_src bash
docker build -t mademanifest:latest src
#+end_src

Run it locally:

#+begin_src bash
docker run --rm -p 8080:8080 mademanifest:latest
#+end_src

The image installs Swiss Ephemeris at the canonical locations:

- =/usr/local/lib/libswe.so=
- =/usr/local/share/swisseph/= – =.se1= data files
- =SE_EPHE_PATH=/usr/local/share/swisseph=

Phase 12 removed the legacy =/app/canon= JSON directory: the
trinity HTTP path consumes only the compiled-in canon constants in
=pkg/canon/constants.go=, so the image no longer ships any canon
JSON files.

The Dockerfile copies the compiled library and headers directly rather
than running =make install=, because the patched Swiss Ephemeris
install target still shells out through =sudo=.

** HTTP Service (Kubernetes)

Tag and push the image to a registry your cluster can pull from (or
use =kind load docker-image= for local clusters), then apply the
kustomize base:

#+begin_src bash
kubectl apply -k src/deploy/kubernetes
kubectl rollout status deployment/mademanifest
kubectl port-forward service/mademanifest 8080:80
#+end_src

*** Production Cluster (Cyso Managed Kubernetes)

The =src/deploy/cyso/= overlay drives deploys to the Cyso cluster
that hosts production (project =Trinity= / cluster =trinityengine=
/ region =ams2=).  It is a thin overlay over =src/deploy/kubernetes=
that sets the deploy namespace and substitutes the image
reference; everything cluster-specific lives there so the base
manifests stay reusable.

[[file:src/deploy/cyso/README.org][src/deploy/cyso/README.org]] is the operator manual for the
overlay; the section below is the abridged version.

**** One-time setup

1. *Save the Cyso kubeconfig.*  Cyso issues kubeconfigs that
   expire after 24 hours.  Save the latest one from the cluster
   owner to a path of your choosing, e.g.:

   #+begin_src bash
   mv ~/Downloads/emk-cluster-*-trinityengine-limited-lifetime.yml \
      ~/.kube/cyso-trinity.yaml
   chmod 600 ~/.kube/cyso-trinity.yaml
   #+end_src

   When the token expires the deploy script fails on the first
   =kubectl= call; request a fresh kubeconfig (or set up the
   permanent service-account token sketched at the end of the
   overlay's README).

2. *Log in to the image registry.*  Default registry is
   =docker.io/informatimago=; one =docker login= populates the
   local Docker keychain so subsequent pushes do not prompt:

   #+begin_src bash
   docker login docker.io
   #+end_src

   Override with =REGISTRY=ghcr.io/<org>= (after a matching
   =docker login=) when pushing somewhere else.

**** Deploy a new version

Single command — every invocation builds, pushes a uniquely-tagged
image, applies the overlay, and smoke-tests the rollout:

#+begin_src bash
KUBECONFIG=~/.kube/cyso-trinity.yaml src/deploy/cyso/deploy.sh
#+end_src

The script:

1. Validates preconditions (=docker=, =kubectl=, =KUBECONFIG=,
   registry credentials).
2. Computes a unique tag from =canon.EngineVersion= +
   =git rev-parse --short HEAD= (+ =-dirty= if =src/= has
   uncommitted changes).
3. Builds the image with =docker buildx build --platform
   linux/amd64= (cross-compiles transparently from Apple Silicon).
   The build runs the canon-pinned IANA tzdata 2026a chain
   (SHA-512 + PGP verification + =zic= compile) and the
   =canon.AssertTZDBVersion= boot gate.
4. Tags + pushes to =${REGISTRY}/mademanifest-engine:${TAG}=.
5. Renders kustomize in a temp directory (committed YAML stays
   clean), runs =kubectl apply -k=, waits for the rollout
   (default 5-minute timeout).
6. Smoke-tests via =kubectl port-forward= against =/healthz= +
   =/version=, cross-checking =engine_version= against the local
   tag so a stale image cache surfaces immediately.

Common flags: =--no-build= (re-tag/push/apply only), =--no-push=
(local mirror), =--no-apply= (pre-warm the registry), =--no-smoke=
(skip the post-rollout smoke).

**** Browser test client against Cyso

A second helper opens =src/scripts/client.html= against the
already-running Cyso engine — useful for ad-hoc browser testing
without a local cluster:

#+begin_src bash
KUBECONFIG=~/.kube/cyso-trinity.yaml src/deploy/cyso/dev-client.sh
#+end_src

The helper sets =TRINITY_DEV_CORS=1= on the live deployment and
waits for the rollout, opens =kubectl port-forward svc/mademanifest
8080:80=, launches the browser at =client.html= (which already
defaults to =http://127.0.0.1:8080=), and on =Ctrl-C= tears down
the port-forward AND re-disables CORS so the deployment returns
to its production posture.  This is the developer path; an actual
end-user front-end requires an Ingress + cert-manager + auth in
front of the cluster (out of scope for this overlay until the
front-end architecture is pinned).

*** Interactive local-k8s dev loop (scripted)

Three helper scripts wrap the manual steps so you can spin a
throwaway cluster up, exercise it with =curl=, and tear it down.
Requires =docker=, =kind=, =kubectl= on =PATH=.

Terminal 1 – bring the service up and open a port-forward
(foreground; =Ctrl-C= deletes the cluster it created):

#+begin_src bash
src/scripts/k8s-local-up.sh
# options: --keep-cluster, --port 9090, --cluster my-cluster
#+end_src

Terminal 2 – send =GET /healthz= and =POST /manifest= through the
port-forward:

#+begin_src bash
src/scripts/k8s-local-test.sh
# overrides: URL=http://127.0.0.1:9090 FIXTURE=path/to/input.json
#+end_src

If a run crashed mid-way and left a cluster behind (or you used
=--keep-cluster= earlier), the down script removes it:

#+begin_src bash
src/scripts/k8s-local-down.sh
#+end_src

Notes:
- The up script reuses a pre-existing =trinity-dev= cluster instead
  of deleting it, so iterative runs stay fast.
- Image tag defaults to =mademanifest:dev=; the kustomize base's
  pinned image reference (digest or tag) is rewritten in-line
  before apply (no temp overlay directory).
- The test script writes the full =/manifest= response to a
  =/tmp/mademanifest-response-*.json= file and prints the path.

*** Browser Test Client (form + result tables)

The repository ships a single-file HTML+JavaScript test client at
[[file:src/scripts/client.html][src/scripts/client.html]] that pairs with the same port-forward.
It renders a form pre-populated with the canonical Schiedam
baseline, builds the Trinity input JSON, sends =POST /manifest=,
and renders the response as nested HTML tables (input_echo,
astrology system + angles + house cusps + 13 objects, human_design
system + activations + channels + centers + type / authority /
definition / profile + incarnation_cross, gene_keys), with
download buttons for the input and output JSON files.

CORS is opt-in for security.  The engine ships with CORS *off* by
default (production-safe) and exposes a =--dev-cors= flag (and a
matching =TRINITY_DEV_CORS=1= environment variable for k8s
deployments) that flips on the wildcard
=Access-Control-Allow-Origin: *= header plus the OPTIONS preflight
short-circuit needed for browser fetches with
=Content-Type: application/json=.  Wildcard CORS in front of an
authenticated ingress is gratuitous attack surface even on a
stateless calculation service, so production deployments must
leave both knobs unset.

Workflow with the kind cluster:

#+begin_src bash
# Terminal 1: brings up kind, applies kustomize, sets
# TRINITY_DEV_CORS=1 on the deployment, opens port-forward.
src/scripts/k8s-local-up.sh

# Browser:
open src/scripts/client.html
#+end_src

The page targets =http://127.0.0.1:8080= by default; the "Base URL"
field at the top of the form lets you point it at a different
host or port (e.g. =http://127.0.0.1:9090= when the up script was
invoked with =--port 9090=).

Workflow without kind (direct binary):

#+begin_src bash
make -C src compile
SE_EPHE_PATH=/usr/local/share/swisseph PORT=8080 \
    src/mademanifest-engine/mademanifest-engine --dev-cors

# Browser:
open src/scripts/client.html
#+end_src

The =--dev-cors= flag prints a startup warning so it is obvious
when the engine is running with CORS enabled.

Workflow against a Docker container (dev image):

#+begin_src bash
docker run --rm -p 8080:8080 \
    -e TRINITY_DEV_CORS=1 \
    mademanifest-engine:itest

# Browser:
open src/scripts/client.html
#+end_src

The Phase 13 NetworkPolicy in =src/deploy/kubernetes/networkpolicy.yaml=
still applies in any Kubernetes deployment: even with =--dev-cors=
on, only the ingress controller can reach the service from inside
the cluster.  The CORS opt-in only affects which browser origins
can drive an already-reachable instance.
