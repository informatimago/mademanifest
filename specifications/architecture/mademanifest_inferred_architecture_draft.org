* Purpose of this draft

This document infers a plausible high-level data model and inter-module data flow
from the following three PDFs:

- /Hybrid Architecture Specification/
- /System Flow/
- /One Pager – Deterministic Narrative Generation System/

It is not a statement of canon. It is a reconstruction intended to make the
architecture more operational by spelling out likely JSON contracts, orchestration
behavior, and missing questions.

The PDFs define responsibilities and boundaries, but mostly at the conceptual
level. They do not define concrete API contracts. Therefore, all schemas below are
*inferred* and should be validated with MadeManifest before implementation.

* Executive summary

The specified architecture is *not* a purely linear pipeline in the strict
stream-processing sense.

It is better described as:

- a deterministic generation pipeline for the main semantic flow:
  Input -> Profile -> Selection -> Blueprint -> Assembly -> Optional Smoothing -> Delivery
- with one sidecar / cross-cutting subsystem:
  Audit Layer
- and with at least one repository-style dependency:
  Content Service / Block Library
- plus a likely top-level application driver / orchestration layer, even though it
  is not explicitly named in the PDFs

In particular:

- /Profile Service/, /Selection Service/, and /Blueprint Service/ look pipeline-like.
- /Assembly Service/ does not merely consume previous output; it likely *drives*
  retrieval from the /Content Service/ and may also emit detailed audit events.
- /Smoothing Service/ is subordinate to the deterministic assembly result and must
  fall back to the deterministic output if validation fails.
- /Delivery Integration Layer/ is more of an adapter / gateway than a pure
  transformation stage.
- /Audit Layer/ is not in series: it runs alongside all steps.

Therefore the architecture is best modeled as a /coordinated pipeline with service
calls/, not as a single simple chain of transforms.

* System boundary inferred from the PDFs

The system starts at *validated input data* and ends at *delivery-ready outputs*.

Upstream of the system, and therefore out of scope:

- frontend / checkout
- user interaction
- raw input validation
- payment / subscription / order capture

Downstream of the system, and therefore out of scope:

- email scheduling
- sending
- retry policy
- external delivery execution

This strongly suggests that the internal boundary begins with something like a
validated generation request and ends with a delivery payload, or with a stored
generated output plus delivery-ready rendering.

* Inferred top-level orchestration

The PDFs name modules, but do not explicitly name the process that coordinates
them. In practice, one of the following must exist:

- an application service
- a job runner
- a workflow engine
- a backend endpoint handler
- a CLI command for deterministic generation

This orchestration layer probably:

1. accepts a generation request,
2. loads or computes profile data,
3. asks Selection for the relevant subset,
4. asks Blueprint for the structure for a given product / format,
5. asks Assembly to build the narrative, with Assembly itself querying the block library,
6. optionally invokes Smoothing,
7. persists output and trace,
8. optionally invokes Delivery Integration,
9. returns identifiers and artifacts.

Without such an orchestrator, the specified modules cannot be combined cleanly.

* Inferred high-level data model

The PDFs imply at least the following conceptual entities:

- /Validated Input/
- /Profile/
- /Selection/
- /Blueprint/
- /Content Block/
- /Narrative Assembly/
- /Smoothed Narrative/
- /Delivery Payload/
- /Audit Trace/

A plausible object graph is:

- one /Generation Request/
  - produces one /Profile/
  - one /Selection/ for a given product
  - one /Blueprint/ for a given format
  - one or many /Assembled Output Units/
  - optionally one /Smoothed Output/
  - one /Delivery Payload/
  - one /Audit Trace/

* Relation to the known MadeManifest PoC / Golden Case

Based on the PoC and prior project context, the /Profile/ probably contains more
than generic marketing variables. It likely includes deterministic variables derived
from a birth/input record, such as some combination of:

- normalized birth metadata,
- planetary or ephemeris-derived values,
- Human Design activations,
- Gene Keys or related derived structures,
- profile- or chart-level summary variables,
- derived categorical labels used by selection rules.

This matters because the downstream modules appear to treat variables as /already
computed meaning/, not as free text.

That suggests a profile JSON more like a structured semantic snapshot than a simple
user attributes object.

* Inferred control-flow shape

** Not purely serial

At least the following call graph is plausible:

#+begin_example
Generation Driver
  -> Profile Service
  -> Selection Service
  -> Blueprint Service
  -> Assembly Service
       -> Content Service (repeated lookups)
  -> Smoothing Service (optional)
  -> Delivery Integration Layer
  -> Audit Layer (parallel events throughout)
#+end_example

So the chain shown in the PDFs is correct at the headline level, but operationally:

- /Assembly Service/ is a driver of /Content Service/
- /Generation Driver/ is the driver of all major steps
- /Audit Layer/ is orthogonal
- /Delivery Integration Layer/ is a gateway, not just a transformation stage

** Possible alternative orchestration

Another implementation option would be event-driven orchestration:

- profile-generated event
- selection-computed event
- blueprint-resolved event
- assembly-requested event
- output-generated event
- smoothing-requested event
- delivery-payload-created event

But the PDFs do not suggest asynchronous architecture explicitly. Their tone suggests a
deterministic request-driven workflow, likely synchronous or job-based.

* Missing technical details inferred from the gaps

The PDFs leave several implementation-critical items unspecified:

** Identity and addressing

- How are users / subjects identified?
- Is generation based on a user id, an order id, or a one-shot input payload?
- Can the same subject have multiple profiles under different canon versions?

** Versioning

- Is every service versioned independently?
- Is there one global canon version?
- Are content, blueprint, and selection rules versioned separately?
- What is the regeneration key? input hash + product + versions?

** Schema ownership

- What is the canonical profile JSON schema?
- What are the variable naming conventions?
- Are nested structures allowed as selection targets, or only flattened keys?
- Are there typed variables, e.g. scalar, enum, range, composite, list?

** Section mapping

- Does Selection choose variables only, or does it also assign them to sections?
- Does Assembly map variables to sections via rules?
- Where do narrative priorities live?

** Content model

- Are blocks single-variable only, or can one block depend on several variables?
- Are variants purely stylistic, or semantically distinct but equivalent?
- How are transitions represented?
- Are placeholders permitted inside blocks?

** Smoothing validation

- What exact validator decides that meaning has not changed?
- Is smoothing reproducible?
- Is the smoothed output versioned independently?
- Is the original deterministic output always stored alongside the smoothed one?

** Rendering and delivery

- Who generates email subject lines?
- Who renders HTML and plain text?
- Who renders report PDF / DOCX / HTML?

* My current best interpretation of the architecture

The architecture is a /deterministic semantic assembly backend/.

It is *not* yet a full platform specification.

Concretely:

- The semantic core is well enough described at the concept level.
- The integration contracts are not yet adequately specified for implementation.
- The engine boundary can be made clean only if MadeManifest confirms:
  - the input schema,
  - the profile schema,
  - selection semantics,
  - blueprint schema,
  - content block retrieval model,
  - assembly rules,
  - smoothing validation,
  - delivery payload expectations.

* Questions for MadeManifest

Below is the question list to carry into the kickoff or canon-package review.

** Scope and boundary

1. Can you confirm the exact boundary of the engine implementation:
   does it end at deterministic assembled output, at validated smoothing, or at
   provider-specific delivery payload generation?

2. Is /Delivery Integration Layer/ part of the engine implementation scope for this
   phase, or is only the deterministic generation core in scope?

3. Is frontend / checkout permanently out of scope for this engagement, or just out
   of scope for the first implementation phase?

** Input contract

4. What is the canonical JSON schema for the validated input entering the system?

5. Which input fields are guaranteed to be validated upstream, and which invariants
   may the backend rely on without revalidation?

6. Is generation addressed by raw input payload, by subject id, by order id, or by
   another identifier?

** Profile schema

7. What is the canonical schema of the generated Profile object?

8. Are variables addressed as hierarchical JSON paths, flattened keys, or both?

9. Which variables are considered first-class selection targets?

10. Are all downstream semantics expected to come exclusively from the Profile, with
    no reinterpretation in later modules?

** Selection semantics

11. Does Selection only choose variables, or does it also assign them to sections /
    slots / narrative roles?

12. What deterministic policy governs prioritization and cap limits when too many
    variables are candidates?

13. Should Selection return variable references, copied values, scored candidates,
    or a richer explanation object?

** Blueprint schema

14. What is the canonical blueprint representation?
    Sections only, or also slots, length targets, repetition rules, and narrative arcs?

15. For email journeys, does the blueprint include cadence metadata, or is cadence
    entirely external to the engine boundary?

16. Are blueprints global per product, or can they vary by locale, customer segment,
    or profile class?

** Content block model

17. What is the canonical content block schema?

18. Are blocks keyed by single variable + slot only, or can they depend on compound
    conditions involving multiple variables?

19. How are content variants selected deterministically?

20. Are transitions stored as ordinary blocks, as blueprint elements, or generated by
    assembly rules?

21. Are blocks plain text only, or may they contain placeholders / templates?

** Assembly rules

22. What exact data does Assembly receive as input:
    full Profile, Selection result, Blueprint, content references, or all of the above?

23. Where is variable-to-section mapping defined: Selection, Blueprint, or Assembly?

24. What deterministic tie-break rules are used when several valid blocks or variants
    exist for the same slot?

25. Should Assembly emit a structured intermediate representation in addition to final
    text, so outputs remain decomposable and auditable?

** Smoothing

26. Is Smoothing in scope for phase 1, or should the first implementation deliver only
    the deterministic assembled output?

27. What validator decides whether smoothing preserved meaning?

28. If smoothing passes, does the smoothed output become the primary delivered output,
    while the deterministic one remains the canonical source of truth?

29. Is smoothing required to be reproducible, and if so, how is reproducibility
    enforced?

** Delivery and rendering

30. What exact delivery-ready payloads are expected for the first supported output
    types?

31. Who owns rendering into HTML / plain text / report formats?

32. Are delivery provider adapters expected now, or should the engine output an
    internal normalized representation only?

33. What external references must be stored after handoff to delivery systems?

** Persistence, audit, and regeneration

34. Which stores are authoritative for:
    profile,
    output,
    delivery references,
    and audit trace?

35. Is the audit log expected to be append-only?

36. What minimum data must be captured to reconstruct an output exactly?

37. What combination of input hash, product id, and version ids defines output identity?

38. On regeneration after canon or content changes, should prior outputs remain frozen
    and separately addressable?

** Operational mode

39. Should the implementation be exposed as:
    a library,
    a CLI,
    an internal API,
    or some combination of the three?

40. Is the expected workflow synchronous request/response, queued job processing, or
    batch generation?

* Closing note

The module decomposition in the PDFs is conceptually clear and coherent, but it is
still one level above implementable contracts. The main hidden complexity lies in
the places where the PDFs remain abstract:

- exact JSON schemas,
- deterministic tie-break rules,
- ownership of section mapping,
- content block addressing,
- smoothing validation,
- and delivery payload ownership.

Those are the points that should be pinned before implementation begins.
