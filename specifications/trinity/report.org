* Critical Ambiguities And Blockers (RESOLVED 2026-04-28)
** A1. Timezone Source Version — RESOLVED (D20)
- Documents (folded back):
  - Document 03 §3.2 — pinned IANA tzdb release
  - Document 08 §4.3 and §6.4 — source-stack pin requirement
  - Document 12 D20
- Ruling:
  - The authoritative IANA Time Zone Database release for Trinity Engine v1 is *2026a*.
  - The implementation MUST NOT rely on the tzdata embedded in the Go runtime unless that embedded release is confirmed to match 2026a.
  - Runtime or operating-system default timezone data is non-authoritative wherever it can affect canonical UTC conversion.
- Implementation status:
  - =pkg/canon/sources.go: TZDBVersion= updated from working-assumption =2023c= to canon =2026a=.
  - The build still imports =time/tzdata= for portability; an infrastructure follow-up will vendor IANA 2026a explicitly (e.g. a generated tzdata package or sidecar TZ_DIR mount) and add a startup assertion verifying a 2026a-sentinel DST rule.

** A2. House Interval Ownership — RESOLVED (D21)
- Documents (folded back):
  - Document 05 §5.6 — house placement rule
  - Document 06 §3.4 — boundary semantics
  - Document 12 D21
- Ruling:
  - House intervals are *cusp-inclusive* and *next-cusp-exclusive*.
  - A longitude exactly equal to cusp N belongs to House N; a longitude exactly equal to the next cusp belongs to the next house.
  - The House 12 → House 1 wrap follows the same rule.
  - No epsilon, midpoint, dual-inclusive, or library-default ownership rule is allowed.
- Implementation status:
  - =pkg/trinity/astro/house.go::HouseFor= already implements this exactly. Comments updated from "working assumption" to RESOLVED.

** A3. Design-Time Final Timestamp Selection — RESOLVED (D22, *implementation corrected*)
- Documents (folded back):
  - Document 05 §6.4.5–6.4.7 — bisection stop conditions
  - Document 05 §11.2 — emission rule
  - Document 07 §10.3–10.4 — output formatting
  - Document 10 — fixture exactness
  - Document 12 D22
- Ruling:
  - When the design-time bisection stops because the remaining interval width is < 1 second, the canonical Design time is the *lower bound* of the final search interval.
  - The emitted =design_time_utc= uses whole-second precision by *truncation* of that lower bound.
  - No midpoint, nearest-second, ceiling, or alternate adjustment is allowed.
- Implementation status:
  - The previous implementation returned the bracket midpoint at the width-stop. =pkg/hd/calc/designtime.go= has been corrected: the width-stop and the (pathological) max-iterations exit now both return =lower=.
  - =output.DesignTime.MarshalJSON= already truncates to whole seconds and is unchanged.
  - The Schiedam =design_time= oracle may shift by up to one whole second under the corrected solver; the regression sentinel will need re-baseline if the truncated value crosses a second boundary.

** A4. Error Message Comparison — RESOLVED (D23)
- Documents (folded back):
  - Document 07 §9.5
  - Document 10 §6.3
  - Document 12 D23
- Ruling:
  - Canonical error fixture comparison is based on =error_type= and =error_code= (plus required envelope shape and metadata).
  - The =message= field must be present where required, but its exact wording is not authoritative for fixture exactness unless a later canon revision introduces a formal message catalogue.
- Implementation status:
  - The validator, error envelope, and golden runner already implement type-only comparison with non-empty message. Comments updated to RESOLVED.

* Significant Ambiguities (RESOLVED 2026-04-28)
** A5. =invalid_input= vs =unsupported_input= Boundary — RESOLVED (D24)
- Documents (folded back):
  - Document 04 §6.2 and §11
  - Document 07 §9.4
  - Document 10 §9.3
  - Document 12 D24
- Ruling:
  - Malformed payloads or values that violate the canonical input contract are =invalid_input=.
  - Structurally valid payloads that fall outside v1 supported scope are =unsupported_input=.
  - The v1 input boundary is strict: minute-precision birth time, one person per payload, no optional canonical input fields, no silent coercion / repair / normalisation inside the engine.
  - Multi-person payloads are =unsupported_input=.
- Implementation status:
  - =pkg/trinity/input/validator.go= already classifies on this boundary. Comments updated to RESOLVED.

** A6. IANA Link / Alias Names — RESOLVED (D24)
- Documents (folded back):
  - Document 04 §6.3
  - Document 03 §3.2
  - Document 12 D24
- Ruling:
  - Accepted timezone identifiers are restricted to canonical IANA Area/Location identifiers from the pinned tzdb release.
  - Aliases and link names (e.g. =US/Eastern=) are not accepted unless introduced by a later canon revision.
  - No silent canonicalisation occurs inside the engine. Upstream systems may canonicalise inputs before calling, but that is outside the engine boundary.
- Implementation status:
  - The validator already rejects abbreviations, the documented link prefixes, and any identifier =time.LoadLocation= cannot resolve. Comments updated to RESOLVED. A follow-up will replace the prefix list with a curated zone.tab when the vendored tzdata 2026a is available.

** A7. Golden Fixture Bootstrap — RESOLVED (D26)
- Documents (folded back):
  - Document 10 Appendix A
  - Document 12 D26
- Ruling:
  - The first golden fixture set must not become self-validating.
  - Candidate expected results may be generated from a canon-compliant implementation, but generation by the implementation is necessary and *not sufficient*.
  - Before approval, fixtures must be reviewed against the governing canon — including independent review of ambiguity-prone areas: timezone-sensitive cases, sign / house / gate / line boundaries, design-time sub-second stop behaviour, error classification edge cases.
  - External tools may be used for sanity checking but are not the golden oracle.
- Implementation status:
  - All Phase-N regression sentinels (astrology, design-time, activations, structure, gene_keys) carry comments updated to reflect D26: cross-checks are sanity checks, not oracles; canon-owner approval is required before the freeze is authoritative.

* Potential Conflict Or Tension Points
** C1. "Byte For Byte" Exactness Versus Partially Unpinned Serialization
- Source context:
  - Earlier email language emphasizes exact expected output "byte for byte."
  - Output Specification focuses on canonical structure, field presence, ordered arrays, and pinned numeric/date formatting.
- Tension:
  - The package does not fully pin object member order or a canonical JSON serializer.
  - Document 07 explicitly allows some serialization variation such as trailing zeros depending on implementation serialization.
- Assessment:
  - If review is semantic JSON comparison, this is fine.
  - If review is literal byte-level comparison, this remains under-specified.
