* Objectif

Définir un plan de QA exécutable pour le moteur =mademanifest-engine=
(=v1.0.0-trinity=) couvrant : revue de code, tests unitaires, fuzzing,
tests d'intégration, cas limites, et couverture.  Le plan cible la
surface trinity-only restante après la Phase 12 (un seul binaire
runtime : =cmd/httpserver=) et les 71 fichiers Go (~24 fichiers de
test) du module.

Le plan est organisé en lots indépendants pour pouvoir être traité en
parallèle ou séquentiellement.  Chaque lot indique :
- la portée (paquets / fichiers concernés) ;
- les sorties attendues (rapport, patch, fixture, métrique) ;
- les critères d'acceptation (=GO=/=NO-GO=).

* Lot 1 — Revue de code

** Objectifs

1. Vérifier l'alignement entre le code et les Documents 00..12 du
   canon Trinity (=specifications/trinity/=).
2. Détecter les écarts par rapport aux invariants canoniques :
   - ancrage mandala 277.5°, intervalle start-inclusive / end-exclusive ;
   - ordre canonique des signes / portes / centres ;
   - bisection backward-only et purement déterministe ;
   - JSON canon-fence (clés whitelistées en sortie).
3. Auditer la sécurité Phase 13 (non-root, =readOnlyRootFilesystem=,
   capacités, =NetworkPolicy=, HPA) sur les manifests
   =deploy/kubernetes/=.

** Méthode

- Revue par paquet, top-down : =canon= → =astronomy= → =ephemeris/sweph=
  → =hd/calc= → =hd/structure= → =trinity/{input,astro,hd,genekeys,output}=
  → =httpservice= → =cmd/httpserver=.
- Pour chaque paquet : grille de relecture (cf. annexe A) cochée.
- Croiser les commentaires de code avec les ambiguïtés A1..A8 du
  registre — toute hypothèse de travail doit pointer vers
  =version-pins.org=.

** Livrables

- =src/doc/qa-review-2026-05.org= : un point par finding
  (sévérité × paquet × ligne × référence canon).
- Patches de remédiation soumis sur la branche QA.

** Critères d'acceptation

- 0 finding =Critical= ouvert.
- Tous les findings =Major= ont un ticket ou un patch.
- Chaque finding référence un Document 0X ou un A* du registre.

* Lot 3 — Fuzzing / Property-based testing

** Cibles prioritaires

1. =pkg/trinity/input.Validate= : entrée JSON adverse, dérivée des
   règles Document 04.
   - Corpus seed : tous les fichiers =input.json= dans
     =src/golden/trinity/{invalid_input,unsupported_input,incomplete_input,valid_*}/=.
   - Fuzz function =FuzzValidate(f *testing.F)= utilisant
     =testing/F.Fuzz= (Go 1.18+).
   - Invariants : (a) jamais de panic ; (b) =error_type ∈
     {invalid_input, unsupported_input}= sur cas rejet ; (c)
     =ContentType= JSON imposé.

2. =pkg/hd/calc.MapToGateLine= : fuzz sur =longitude ∈ [0,360)=,
   propriétés :
   - jamais de panic ;
   - =gate ∈ [1,64]=, =line ∈ [1,6]= ;
   - monotonicité par segments : =MapToGateLine(λ)= ne change pas dans
     un intervalle ouvert canonique ;
   - bord : =MapToGateLine(start) != MapToGateLine(end)= sur chaque
     frontière canonique (start-inclusive / end-exclusive).

3. =pkg/hd/calc.SolveDesignTime= : property-based avec entrées
   =personalityTime= aléatoires sur 1900..2100, propriété
   =SunLongitude(SolveDesignTime(t)) ≈ SunLongitude(t) − 88°= modulo 360°
   à ε près.

4. =pkg/astronomy= : fuzz =LocalToUTC= avec fuseaux IANA aléatoires
   (subset de =zonetab=).
   - Propriété : aller-retour =UTCToLocal(LocalToUTC(t,zone),zone)=
     idempotent hors transitions DST.

5. =pkg/httpservice= : fuzz sur le corps de =POST /manifest= ;
   propriété : pas de 5xx, seulement 200 / 4xx, et schéma d'enveloppe
   d'erreur respecté (Document 04, A4).

** Outils

- =go test -fuzz= natif, budget 10 min/cible en CI nightly,
  60 min/cible en pre-release.
- =gopter= pour les propriétés algébriques (SolveDesignTime).
- Corpus persistant sous =src/mademanifest-engine/testdata/fuzz/=.

** Critères d'acceptation

- 0 panic, 0 5xx découvert.
- Corpus de régression élargi avec chaque crash découvert.
- Rapport =qa-fuzz-2026-05.org= : durée, exécutions/s, crashs uniques.

* Lot 4 — Tests d'intégration

** Périmètre

Le harnais =integration/= existe en trois saveurs (build tags) :
=integration_local=, =integration_docker=, =integration_k8s=.  Il
exécute :
- Trinity Golden Pack (=pkg/golden= sur =src/golden/trinity/=) ;
- contrats HTTP (=AssertHTTPContract=) ;
- baseline Schiedam (Phase 4..8) ;
- =SE_NODE_POLICY= env immunity ;
- forme du déploiement durci (=AssertHardenedDeploymentShape=) ;
- =AssertNetworkPolicyEnforced=, =AssertHPAScalesUnderLoad= (gated).

** Plan d'exécution

1. =make test-integration-local= sur Go 1.22 + Swiss Ephemeris 2.10.03
   pinné (timeout 120 s).
2. =make test-integration-docker= sur image
   =mademanifest-engine-dev= (timeout 600 s).
3. =make test-integration-k8s= sur kind, kustomize overlay incluant
   =SE_NODE_POLICY=true= (timeout 900 s).
4. Activation des deux gates :
   =TRINITY_NETWORK_POLICY_ENFORCED=1= et =TRINITY_LOAD_TEST=1=.
5. Vérification de la sortie =GET /version= : présence de
   =ephe_path_resolved=, absence de toute fuite dans =/healthz=.

** Extensions à ajouter

- Cas =GET /healthz= sous charge : 100 RPS pendant 60 s, taux d'erreur
  < 0.1 %.
- =POST /manifest= avec =Content-Type= manquant ou erroné → 415,
  =error_type=invalid_input= (Phase 10).
- Corps > =MaxRequestBodyBytes= (1 MiB) → 413,
  =error_type=unsupported_input=.
- JSON malformé → 400, =error_type=invalid_input= (Phase 2).
- Trois exécutions consécutives du Pack Golden : sortie
  bit-identique (déterminisme Phase 9).

** Critères d'acceptation

- Les trois harnais (local, docker, k8s) verts en CI.
- Rapport déterministe sur 3 exécutions consécutives.
- =trinity-nightly.yml= vert sur les 7 derniers jours.

- Corps vide ; corps = =null= ; corps = =[]=.
- Encodage =Content-Encoding: gzip= non supporté (rejet propre).
- =OPTIONS /manifest= et =HEAD /healthz=.
- En-tête =Content-Type: application/json; charset=utf-8= (doit passer).

- =authority= dégénérée : 0 centre défini → fallback documenté.
- =profile 6/2= et =6/3= : transitions de pression (vie 1 / vie 2).

** Critères d'acceptation

** Stratégie

- =make test-coverage= produit =coverage.txt= ; pousser en CI un
  =coverage-report.html= comme artefact.
- Seuils par paquet :
  | Paquet                          | Seuil min |
  |---------------------------------+-----------|
  | =pkg/canon=                     | 95 %      |
  | =pkg/hd/calc=                   | 95 %      |
  | =pkg/hd/structure=              | 95 %      |
  | =pkg/trinity/input=             | 95 %      |
  | =pkg/trinity/output=            | 90 %      |
  | =pkg/trinity/{astro,hd,genekeys}= | 90 %    |
  | =pkg/httpservice=               | 90 %      |
  | =pkg/golden=                    | 85 %      |
  | =pkg/astronomy=                 | 90 %      |
  | =pkg/ephemeris=, =pkg/sweph=    | 70 % (cgo)|
  | =cmd/httpserver=                | n/a (couvert par intégration) |

- Couverture combinée (unit + intégration) via =-coverpkg=./pkg/...=
  injecté dans le binaire d'intégration.

** Anti-objectifs

- Ne pas viser 100 % : les chemins =log.Fatalf= au boot
  (=SelfCheck= panic, =ValidateEphePath=) sont volontairement
  non-couverts par les unit tests, mais doivent l'être par un test
  d'intégration =refuses to start=.

** Critères d'acceptation

- Seuils respectés au global et par paquet.
- Aucun fichier sous 60 % sans dérogation documentée.
- Diff de couverture sur la PR : pas de régression > 1 pp.

* Annexe A — Grille de revue de code (par paquet)

1. Tous les chemins d'erreur exposent un =error_type= compatible A4 ?
2. Aucune comparaison de flottants par ==== dans les chemins canoniques
   (utiliser =math.Abs(a-b) < ε=) ?
3. Aucune dépendance à l'ordre d'itération des =map= dans la sortie ?
4. Pas de =time.Now()= ou =rand= dans le chemin de calcul (Phase 9) ?
5. Pas de variable d'environnement lue en dehors du boot ? (=SE_*= en
   particulier, doit être inerte après Phase 9)
6. JSON marshaling : seuls les champs whitelistés sont émis ?
7. Tests : présence d'un cas =error_type= explicite par mode d'échec ?
8. Goroutines : aucune goroutine non-jointe dans le chemin
   =POST /manifest= ?
